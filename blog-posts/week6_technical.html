Blocks, Procs, and Lambdas

Blocks

When learning Ruby, a common refrain is that "everything is an object." As it turns out this is not entirely true. One of the rare exceptions is our friend, the block. A block is just a bit of code that can be executed. The reason blocks aren't considered objects is because they can't be saved to a variable, and they don't have all the functionality in an object's repertoire.

As for syntax, a block is denoted by curly braces ({}) or by do...end. Some common methods that are followed by a block are #each, #times, and #collect, but there are a ton of others.

Procs

Let's say you have a block of code that you intend to use several times in a program. Wouldn't it be cool if you could save it to a variable and just use that variable instead of having to write out that bit of code every time you wanted to execute it? Well, that's what procs are for.

Creating procs is very similar to creating other objects. All you have to do is call Proc.new and pass in a block of code. So for example,

double = Proc.new { |i| i * 2 }

[1, 2, 3].map(&double) # => [2, 4, 6]

(the "&" in front of the Proc variable name just converts the proc into a block since that's what #map normally takes)

There's another useful way to call a Proc, and that's with the #call method. Check it,

jamaican = Proc.new { puts "whatta gwan" }

jamaican.call # => "whatta gwan"

Lambdas

With the exception of a couple of small details, lambdas and procs are essentially the same thing. There's a slight difference in how you create new procs and lambdas, as seen here:

jamaican = Proc.new { puts "whatta gwan" }

jamaican = lambda { puts "whatta gwan" }

Other than that, lambdas are aware of the number of arguments that gets passed to it while procs are not. Lambdas will throw an error if they get the wrong number of arguments; procs make do by either ignoring extra arguments or adding nil values if there aren't enough.

And finally, when a lambda returns whatever value it's supposed to return, it passes control back to the method that called it. So if the calling method then returns another value (after the lambda), it will do so unimpeded. Procs on the other hand will return and not pass control back to the calling method. So once the proc returns its value, the calling method does not return anything else that follows the proc, even if it's written in such a way that suggests that a value should be returned.
